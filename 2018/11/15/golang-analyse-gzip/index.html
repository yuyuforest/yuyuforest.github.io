基本的结构体 Writer

定义了一个新的 Writer 结构体，用于进行压缩的处理。

  • Header 是定义于 compress/gzip/gunzip.go 中的结构体，给出了关于压缩文件的元数据。它并不是响应体的首部行。它的成员如下：

  • w 的类型 io.Writer 是一个接口，包括了最基础的写函数，可以将多个字节写入底层的写入器。

    实现了这个接口的类型包括 bytes.Buffer 。

  • level 指压缩级别。

  • wroteHeader 表示是否已写元数据。

  • compressor 用来对原始数据进行压缩。

  • digest CRC-32-IEEE校验。

  • size 未压缩的原始文件的大小。

  • closed 指 Writer 是否关闭。

  • buf 是一个较小的缓冲区，辅助进行对某些元数据从基本类型到字节的转换。参见【—】一节。

  • err 记录 Writer 的操作中出现的错误。

向 io.Writer 写入数据

下面的两个函数，是【写元数据和压缩】中 func (z *Writer) Write(p []byte) (int, error) 的辅助函数。

写字节

要写入的数据 b 的形式为多个字节。b 的长度应不超过 0xffff 。

把原始数据的长度通过 le.PutUint16(z.buf[:2], uint16(len(b))) 记录入 Writer.buf[0:2] ，经缓冲区的转换后写入 Writer.w 。

原始数据 b 写入 Writer.w 。

写字符串

要写入的数据 s 的形式为UTF-8字符串。

由注释可知， s 应符合GZIP要求的格式：

  • 是一个空终止字符串（以 '\0' 结尾的字符串）
  • 是 ISO-8859-1 编码的字符串
  • 如果不是 ASCII 编码的字符串，则需要手动转换成字节序列

把 s 写入 Writer.w 。

把 '\0' 经 Writer.buf[0:1] 缓冲区的转换后写入 Writer.w ，表示字符串的结束。

写元数据，压缩原始数据

检测是否已写元数据，如果没有，则写入。

如果 Writer.compressor 为空，则使用写入器 Writer.w 和压缩级别 Writer.level 来创建一个压缩器。

更新压缩前的大小 Writer.Header.size 和校验码 Writer.Header.digest。

通过 Writer.compressor.Write(p) 把原始数据的压缩结果写入底层的writer。

刷新

刷新 z.compressor ，强制将已压缩、未写入的数据写入底层的writer。

关闭 Writer

在关闭之前，将校验码和未压缩的文件大小写入 io.Writer 。

基本的结构体 gzipResponseWriter

gzipResponseWriter 封装了响应写入器 negroni.ResponseWriter 和gzip压缩写入器 gzip.Writer ，共同进行压缩和响应处理。

wroteHeader 表示首部行是否已编码。

其他结构体

gzipResponseWriterCloseNotifier 封装了 negroni.ResponseWriter 结构体，可调用 func CloseNotify() <-chan bool 函数，因而实现了 http.CloseNotifier 接口。它用于监测底层连接是否已关闭。

handler 包括一个对象池。它是一个可以处理和响应请求的处理器，可以调用 func (h *handler) ServeHTTP(w http.ResponseWriter, r *http.Request, next http.HandlerFunc) 对请求进行压缩相关的处理。（详见【处理器控制压缩】一节。）

通知连接关闭

gzipResponseWriterCloseNotifier 实现了 http.CloseNotifier 接口。CloseNotifier 用于监测底层连接是否已关闭， CloseNotify 负责发出通知。

这个机制可用于在响应未就绪而客户端断开连接的情况下，取消服务器上的长时间操作。

调用 ResponseWriter 的 WriteHeader 方法。wroteHeader 置为 true ，表示已设置首部行属性。

获取控制压缩的处理器

创建一个处理器 handler ，和标准库中原生的gzip写入器 gzip.Writer ，后者放入前者的对象池中。

处理器控制压缩

handler 通过 ServeHTTP 函数对http请求进行处理和响应。它只进行和压缩有关的处理，其他的处理工作交给处理器 next 。

首先检测请求的首部行 r.Header 的 AcceptEncoding 属性是否包括了 encodingGzip 。如果客户端并没有要求 gzip 压缩编码，那么就不进行压缩，把请求和 http.ResponseWriter 传给处理器 next 继续处理。

如果客户端要求 WebSocket 连接，handler 也不进行压缩处理， next 继续处理。

从处理器 handler 的对象池中取出 gzip.Writer ，重置，准备写入新数据。可见， gzip.Writer 的 Reset 方法允许对不同的请求复用一个已分配的缓冲区，而非每次都从头分配一次缓冲区。

创建 negroni.ResponseWriter 和 gzipResponseWriter 。将 gzipResponseWriter 传递给处理器 next 继续处理。剩下的压缩工作或将在 next 中处理。
