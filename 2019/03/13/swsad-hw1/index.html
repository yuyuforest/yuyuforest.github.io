简答题

软件工程的定义

根据IEEE标准610.12，软件工程是将系统化、规范化、可量化的方法应用于软件的开发、操作和维护，即将工程应用于软件，以及对以上方法的研究。

解释software crisis的本质原因、表现，述说克服软件危机的方法

  • 软件危机的本质原因：现有的软件开发方法不足以适应急速增长的计算机性能，软件复杂性的增加引起了许多软件开发和维护的问题。
  • 软件危机的表现：
    • 项目运行超支
    • 项目运行超时
    • 软件效率低下
    • 软件质量低下
    • 软件达不到需求
    • 项目难以管理，代码维护困难
    • 软件从未被交付
  • 克服软件危机的方法：
    • 正确认识计算机软件的内涵。
    • 采用工程项目管理方法实施软件开发的组织管理。
    • 研究新的系统化的、与时俱进的软件工程方法，开发辅助软件项目管理与技术研发的软件工具。

软件生命周期

又称为软件开发过程，是将软件开发工作在时间维度上划分为不同的阶段，以此改进设计、产品管理和项目管理的过程。

主要的开发模型有敏捷开发、瀑布开发、螺旋开发等。

SWEBOK 的 15 个知识域

  1. Knowledge Areas Characterizing the Practice of Software Engineering

    表征软件工程实践的知识领域

    1. Software Requirements 软件需求

      该知识领域要求关注软件需求的启发、协商、分析、规范和验证。软件需求表达了对软件产品的需求和限制。

    2. Software Design 软件设计

      该知识领域覆盖了设计过程和最终产品。设计被定义为两个部分：定义体系结构、组件、接口及其他的系统或部件的特性的过程，和这个过程的结果。软件设计阶段是软件工程生命周期的活动，在这期间软件需求被分析从而产生软件内部结构及行为的描述，这是软件构造的基础。

    3. Software Construction 软件构建

      软件构建指通过结合详细设计、编码、单元测试、集成测试、调试和验证来详细地创建工作软件。

    4. Software Testing 软件测试

      软件测试旨在评估产品质量、识别缺陷以提升产品质量，需要在有限的测试用例集上针对预期行为来动态地检查程序。

    5. Software Maintenance 软件维护

      软件维护包括加强现有功能，调整软件以在新的或变化了的操作环境中运行，以及纠正缺陷。

    6. Software Configuration Management 软件配置管理

      软件系统的配置指硬件、固件、软件或它们的组合的功能特征和/或物理特征。它也可以被视为特定版本的硬件、固件、软件的集合，这些组件根据特定的构建过程来组合，以满足特定目的。

    7. Software Engineering Management 软件工程管理

      软件工程管理涉及规划、协调、测量、报告和控制项目或程序，以确保软件的开发和维护是系统化、规范化、量化的。

    8. Software Engineering Process 软件工程流程

      该领域关注软件生命周期过程的定义、实施、评估、测量、管理和改进。

    9. Software Engineering Models and Methods 软件工程模型和方法

      该领域提出了涵盖多个生命周期阶段的方法。涵盖的主题包括建模、模型类型、分析、软件开发方法。

    10. Software Quality 软件质量

    该领域包括软件质量的基础知识、软件质量管理流程和实际考虑。

    1. Software Engineering 软件工程专业实践

      该领域关注软件工程师为了专业地、负责任地、道德地实践软件工程，必须具备的知识、技能、态度。涵盖了专业性、道德准则、小组动态、沟通技巧等主题。

  2. Knowledge Areas Characterizing the Educational Requirements of Software Engineering

    1. Software Engineering Economics 软件工程经济学

      该领域要求在业务环境中做出与组织的业务目标一致的决策。

    2. Computing Foundations 计算基础

      该领域涵盖了提供软件工程实践必需的计算背景的基础主题。

    3. Mathematical Foundations 数学基础

      该领域涵盖了提供软件工程实践必需的数学背景的基础主题。

    4. Engineering Foundations 工程基础

      该领域涵盖了提供软件工程实践必需的工程背景的基础主题。

简单解释CMMI的五个级别

  • Initial 初始级：无序的、不可预测的过程，缺乏控制，管理是反应式的。
  • Managed 管理级：建立了基本的项目管理过程来跟踪费用、进度和功能特性，制定了必要的过程纪律。
  • Defined 定义级：将软件管理和工程两方面的过程文档化、标准化，所有项目均使用已定义、经批准的标准软件过程来开发和维护软件。
  • Quantitatively Managed 定量管理级：对软件过程和产品测量详细的数据，有量化的理解和控制，并能够在定量的范围内预测性能。
  • Optimized process 优化级：开发过程有量化的反馈，和先进的新思想、新技术一起促使过程持续改进。

用自己语言简述SWEBOK或CMMI（约200字）

CMMI，Capability Maturity Model for Software，软件能力成熟度模型。它为流程定义了5个成熟度级别：初始、管理、定义、定量管理、优化，因而可用于评估组织过程成熟度，在此基础上可用于指导项目、部门或整个组织的流程改进。虽然CMMI起源于软件工程，但它是一个高度概括的框架，所以也可以应用于其他领域，一定程度上可以满足许多企业发展的需求。需要注意的是，它是一个模型，而非具体的标准，关于它的评估，不同的行业有不同的具体的执行方式。
